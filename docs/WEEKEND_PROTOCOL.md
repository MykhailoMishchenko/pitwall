# Протокол гоночного уикенда (закон аналитических сессий)

Статус: v0.2 (2026-07-24). Флоу анализа пиков — раздел ниже — добавлен
после R11 Угорщина: три подряд найденных пользователем дыры в одном
черновике (погода без проверки, апгрейд Aston Martin переведён но не
исследован, логическое противоречие Q1/Q8) показали, что верификация
происходила ПОСЛЕ публикации чернетки, а не до. Раздел «Флоу анализа
пиков» — это чек-лист, обязательный ДО показа любого чернетки пиков.

## Незыблемое

1. Пики — в `journal/` с таймстемпом ДО лока. После лока read-only навсегда.
2. Каждое отклонение от модели (conviction) — отдельная запись с обоснованием
   ДО лока. Тіньова команда фиксируется тем же таймстемпом.
3. Пропущенный дедлайн фиксируется как пропуск. Бэкдейтов не существует.
4. Разбор полётов обязателен после каждого ГП; learnings — append-only.
5. Замыкание цикла: разбор заканчивается сетапом рутин следующего ГП
   (времена сессий — из Jolpica, не хардкод; спринт-формат определять из
   календаря).

## Рутины (контент — TODO до Спа)

- `preview` (Вт–Ср): профиль трассы (SC%, пит-лосс, обгоны — история),
  форма/цены/погода → черновик пиков v0 → `analyses/R<n>-preview.md`
  (флоу анализа пиков — обязателен, см. ниже)
- `refine` (Пт): практики, лонг-раны, деградация (FastF1) → пики v1
  (флоу анализа пиков — обязателен, см. ниже)
- `final-picks` (лок −3ч): финал состава + Predict EV-таблица → журнал →
  отправка → верификация отправки (флоу анализа пиков — обязателен)
- `safety-check` (лок −1ч): подтверждение отправки | пуш пользователю
- `quali-debrief` (Сб): сетка vs прогноз; boost-окно Predict
- `race-debrief` (Вс): скоринг, прогноз vs факт, ошибки, learnings, лига,
  «человек обыграл модель» → `analyses/R<n>-debrief.md`
- `close-cycle` (Вс–Пн): календарь следующего ГП → рутины → отчёт пользователю

## Флоу анализа пиков (обязательный чек-лист, каждый черновик)

Применяется в `preview`, `refine`, `final-picks` — при ЛЮБОМ выпуске или
пересмотре пиков, до того как это увидит пользователь или уйдёт в
`analyses/*.json` / `journal/`.

### Шаг 1 — Сбор сырых данных (до каких-либо выводов)

- **Сравнимая трасса.** Для любого пика, зависящего от типа трассы
  (подиум, поул, темп) — найти ближайшую по характеру трассу ЭТОГО сезона
  и подтянуть по ней реальные данные (Jolpica: квала-разрыв, гоночный
  результат, статус финиша). Не полагаться на память/нарратив вида «эта
  команда обычно сильна на технических трассах» без цифр.
- **Subtext = триггер на исследование, не текст для перевода.** Каждый
  подтекст вопроса из фида Predict — если содержит конкретный факт или
  намёк (апгрейд, стрик, история личных встреч) — сначала WebSearch/
  проверка, потом обоснование пика. Перевод — последний шаг, не первый.
- **Погода и любые цифры из внешнего источника (включая от пользователя)
  — сверяются САМОСТОЯТЕЛЬНО минимум 2 независимыми источниками ДО
  публикации.** Ничего не публикуется на честном слове источника, кем бы
  он ни был.
- **Источник переоценивается дословно, не по памяти.** Если меняю
  собственный более ранний вывод — перечитываю точную цитату
  первоисточника заново, а не пересказ из предыдущей записи журнала.

### Шаг 2 — Черновик пиков

- Каждый пик — с конкретной цитируемой точкой данных, объясняющей, почему
  это релевантно ИМЕННО для этой трассы/раунда. «Рыночный фаворит» без
  привязки к контексту — недостаточное обоснование.
- Conviction по рубрике: **high** — данные сравнимой трассы и текущая
  форма совпадают; **med** — один сильный сигнал без противоречий; **low**
  — рыночный дефолт / разовое событие без независимого сигнала.

### Шаг 3 — Самопроверка согласованности (ОБЯЗАТЕЛЬНО, до показа)

- Пройтись по ВСЕМ вопросам раунда разом: пилот/команда, упомянутые в
  нескольких вопросах — согласуются ли выводы? (Пример пропущенного бага:
  подиум без Хэмилтона в Q1, но «Хэмилтон на подиуме» в Q8 — прямое
  логическое противоречие, которое проверяется парой минут кросс-чтения.)
- Явно выписать 2-3 самых слабых пика (низкая conviction) — до того, как
  их найдёт пользователь.

### Шаг 4 — На пуш-бэк пользователя

- Сначала проверить независимо (источник/данные), потом отвечать.
  Несогласие — валидный исход, если данные его подтверждают.
- Если меняю пик — ответ обязан содержать НОВЫЙ добытый факт, а не
  просто согласие. «Ты прав» без цитаты источника — недостаточный ответ.
